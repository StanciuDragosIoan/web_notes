Oauth2

OAuth 2.0 is the industry-standard protocol for authorization.
OAuth 2.0 focuses on client developer simplicity while providing specific authorization flows for web applications, desktop applications, mobile phones, and living room devices.

What is OAuth2?

OAuth2.0 is a delegated authorisation framework. This means it allows application A to access resources hosted by application B on behalf of a resource owner.

Let's imagine we have a basic application, 'Example App'. Example App (application A) would like to access some of a user's (resource owner's) information currently hosted on Google (application B). The user grants Example App access to some of their information they stored in Google (in this case their basic profile info and email address). However, how does Example App retrieve this information, and how does Google know to allow Example App access? This is the role of OAuth2.0.

Key concepts:

Authentication:
Making sure you are who you say you are.

Authorisation:
Making sure you are only allowed to do what you're supposed to.

OAuth2.0 is a framework for authorisation and not authentication. It allows an application (Example App) access to resources (basic profile info and email address) hosted by another application (Google). Google might handle the authentication component, but this is outside OAuth2.0.

Therefore, OAuth2.0 cares about what you can do, but not who you are.

Basic OAuth2.0

To introduce the main components of the framework, let's employ a popular analogy: visiting a hotel.

We (the client) are visiting a hotel (so we request access to hotel resources and the hotel is their 'owner').
We go to the front desk (this is the authorization server responsible for giving access to resources) and it gives us a key card (this is an access token a resource that the resource server will check to see if we can access certain resources or not).
With the key from the front desk we can access the hotel's gym, sauna or bar (these are the server resources).
The resources will also have scopes of access (saying what we can and cannot do with the resource) for instance we can order a drink at the bar but not sleep on it.
